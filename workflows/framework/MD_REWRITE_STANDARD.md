# 并入 Markdown 文档统一重写规范

## 强制规则

所有准备并入 Skill、知识库、风格库、项目背景或共享发布包的 `.md` 文档，必须先按照本规范重新编写。原始文档不得直接作为正式规则使用。

原始文件只允许放在 `source/` 或项目的 `01_background/raw/` 中作为来源留档；重写后的文件才可以进入 `references/`、`knowledge/`、`examples/` 或 `09_shared_release/`。

## 重写后的标准结构

```markdown
# 文档标题

文档类型：research | style-rule | prompt-rule | case-record | failure-rule | brand-rule
文档 ID：md-YYYYMMDD-0001
版本：0.1.0
状态：candidate | approved | archived
来源：原始文件路径或项目 ID
重写日期：YYYY-MM-DD
重写人：操作人或 Agent
关联 Skill：skill-id

## 1. 目的
## 2. 适用范围
## 3. 执行规则
## 4. 正向规则
## 5. 负面规则
## 6. 参数与示例
## 7. 质量检查
## 8. 限制与待确认项
## 9. 变更记录
```

## 重写要求

1. 删除聊天口吻、重复段落、无结论的过程描述和平台无关的猜测。
2. 将模糊形容词改为可执行的视觉、参数或检查条件。
3. 分离事实、设计师明确判断、Agent 推断和待确认内容。
4. 将正向规则、负面规则、参数、适用范围和失败边界分开。
5. 保留来源路径、原文版本、操作人和日期，确保可追溯。
6. 不得在重写时擅自提高规则优先级；正式 Skill 规则必须经过审核。
7. 重写完成后状态固定为 `candidate`，审核通过后才可变为 `approved`。

## 验收条件

文档只有同时满足以下条件，才能进入候选知识库：标题、类型、ID、版本、来源、日期、关联 Skill 齐全；正文包含目的、适用范围、执行规则、正向规则、负面规则、质量检查和限制；没有未解释的临时命名或互相冲突的规则。
